# 章节开发指引（每章开发必读）

---

## 这是放映用的网页 slides，不是普通 PPT

正在做的是**点击驱动的网页 slides** —— 讲者全屏放映、用方向键 / 点击
推进、对着 `notes.ts` 的提词现场讲（按 `N` 看提词，观众看不到）。
判断每一步做对没有，标准非常朴素：

- **不像普通 PPT** —— 观众感觉画面在"演"，不是在看翻页幻灯（页面中不得包含页眉页脚，突出主视觉元素）
- **看起来舒服** —— 配色、字体、节奏都让人放松，不得出现大量的纯文字、不得出现字体太小的文字
- **有视觉冲击** —— 画面在演事情，不只是文字堆砌，不得一次性全部罗列所有元素，关键元素随进度逐步推进展现

---

## 两种密度 profile：视频式 / 演示 deck 式（开工前先确认）

同一套引擎能跑两种节奏。**开工每章前先确认当前是哪个 profile**（在
`outline.md` 顶部或 Checkpoint Plan 定好）：

| | 🎬 视频式（默认 / 稀疏） | 🎤 演示 deck 式（有限页 / 高密度） |
|---|---|---|
| 适用 | 录屏讲解、不停讲、节拍多 | 现场 / 投屏演讲、页数有限、围着一页讲（面试复盘常用） |
| 一张"幻灯片" | = 一个 step（整屏接管） | **= 一个 chapter**（一页稳定的密集画面） |
| step 语义 | 切到下一个聚焦想法（换场景） | **同一张 slide 上逐步 build**（元素留屏、渐显 / 高亮，不换场景） |
| 一屏信息量 | 1 hero + 1~3 元素 + 大留白 | 图 / 表 / 示意 + 几条要点 + 数据可同屏 |
| 章节 × step | 很多章 × 少步 | 较少章（≈ 目标页数）× 每章 2~5 build |

**deck 式怎么写**（只列与默认不同的点）：

- **step = build，不是换场景**：渲染**一张稳定的密集画面**，用
  `className={step >= N ? "on" : ""}` 让子元素**逐步加上 / 高亮**，已出现
  的留在屏上当上下文 —— **不要** `if (step===N) return <另一个场景>`。
  （引擎天然支持：`App.tsx` 的 scene 按 `chapter` 绑 `key`，章内换 step
  不重新挂载，build 很顺。）
- **放开"一屏一想法 / 大留白 / 全 hero"**：deck 页允许同屏放图 + 标注 +
  几条要点 + 数据；正文用 `--t-h2` / `--t-body` 一档而非全 hero；留白按
  构图收紧。
- **守住不变**：依然**必须有视觉演示**（图 / 图表 / 示意，不是纯 bullet
  墙）；依然**逐步 build**（禁一次糊满整页再不动）；依然**回原始素材抽
  细节**（双源）；**「避免 AI 味」+ token + 真实层级整套照旧** —— 这半套
  才是"不难看"的根，密度高低都要守。
- **主题**：deck 式优先结构 / 编辑型（newsroom / monochrome-print /
  blueprint / swiss-ikb / vintage-editorial），别用大海报型。

> 不确定就是 🎬 视频式。下面所有规则默认按视频式读；命中 🎤 deck 式时，
> 按本节改写「逐步揭示」「内容取舍」「放映基本审美」三节的**密度档位**，
> 其余（视觉演示底线 / 双源 / 避免 AI 味 / 代码红线 / 完工自检）不变。

## 必须用 CSS / SVG / Canvas / JS 大胆绘制视觉演示

> **这是底线。**
>
> 每一章都至少要有 1~2 处"动起来的图 / 演示元素"。
> **整章只有纯文字 = 验收不过 = 回去重做。**

画面感最强的来源 —— 观众**看见**了被讲解的东西在屏幕上演给他看：

- 数字在递增 / 横条在生长 / 排名在交换
- 流程节点依次点亮 / 连线自绘
- 对比被一刀切开 / 聚光灯扫过 / 形状在变形
- 粒子聚拢成形 / 噪声背景流动 / 字符雨下落
- 模拟终端交互
- 模拟 AI 对话窗口
- 模拟文件目录树

**怎么组合发挥都行 —— 但每章必须用，不允许整章纯文字。**

## 逐步揭示，禁止一次全展示

整页内容由**全局 `step` 计数器**驱动 —— 点击空白处或按 → 键推进
一步。设计每一步时心里要默念：**这一步演什么，下一步演什么**。

**最重要的一条 ——「逻辑节拍 = step」**：

> **一个 step = 一个聚焦想法**。当一段内容是"第一是 X、第二是 Y、
> 第三是 Z"这种**清单 / 列表**时，**严禁**一个 step 把 X / Y / Z 全部
> stagger 上来。

正确做法：

- 一项 = 一个 step
- X 只在它自己的 step 里独自亮起
- 讲到 Y 时，X 灰化保留作上下文 + Y 亮起
- 讲到 Z 时，X / Y 都灰化 + Z 亮起

**判断标准**：讲者会一个一个分开讲吗？会 → 必须逐个揭示。

> 🎤 **deck 式**：「逐个揭示」仍然要 —— 但形式是**在同一张密集 slide 上
> 逐步 build**（元素留屏、渐显 / 高亮），而不是每步切一个稀疏整屏。见
> 上方「两种密度 profile」。

---

## 内容取舍：抓重点，不要原文搬运

放映是**讲 + 画**：

- **讲者**（对着 `notes.ts` 提词）负责把信息线性讲清楚
- **画面**负责把节拍重点放大、节奏感拉出来

每个 step 屏幕上只挂这个节拍**最值得放大的 1~3 个东西** —— 一个
hero 标语 / 一个数字 / 一组对比 + 必要的视觉演示。

不要试图把原始素材每个字都搬上去。那是论文阅读，不是放映。

> 🎤 **deck 式**：一张 slide 可以同屏放更多（图 + 标注 + 几条要点 + 数据），
> 但仍是**有结构的密集**，不是把素材整段贴上去；逐步 build 出来，别一次
> 糊满。

---

## 双源：节奏跟 outline，细节回原始素材

> **节奏 / 顺序 / 节拍切分** 跟 **`outline.md` 开发计划** —— **关键顺序不能乱**。
> **画面细节 / 数据 / 引用 / 案例** 回**用户的原始素材**（article / 文档 /
> 笔记等，下称 material）抽。

`outline.md` 已经在每章首段抽了「信息池」做参考。但**实现章节时
也必须回去翻原始素材本章对应段落** —— 那里有比 outline 多得多的
细节（具体数字、引用原话、案例维度、出处时间）。把这些挂到画面上，
让**画面信息密度足够高**。

> **如果你只用了 outline 的字面内容做章节** —— 屏幕等于把提纲打字打了
> 一遍 —— 那就是普通 PPT，不是会"演"的 slides。
>
> 章节实现一定要回原始素材抽细节，不要嫌麻烦。

---

## 字体 / 配色 / 动画 / 留白 —— 放映基本审美

放映时观众离屏幕远（投屏 / 大屏）、注意力浮动，所以：

- **字号要大** —— hero 文字至少 80px 起，远观也能看清
- **留白要多** —— 舞台四边都要让出大留白，画面不要塞满
- **配色要舒服** —— **颜色和字体家族必须用主题 token**（保证换主题不破）；
  字号 / 间距 / 时长这些章节按内容自由发挥（详见下方「代码层最小约束」）
- **动画要舒服 + 炫酷** —— 出现得干净利落，停下来不抢戏；炫酷靠
  **设计巧思**（内容驱动的演示动画），不靠**速度暴力**或**密集闪烁**

> 🎤 **deck 式**：「字号要大 / 留白要多」放宽 —— 正文可用 `--t-h2` /
> `--t-body` 一档而非全 hero、留白按构图收紧；但**字号仍要远观可读**、
> **配色 / 层级仍要舒服**，token 与「避免 AI 味」不放宽。

## 避免 AI 味

AI 生成的网页有几种共有的"视觉指纹"，**全部不要**：

- 紫粉 / 蓝紫对角渐变背景
- 圆角卡片 + 彩色左边框装饰
- 渐变按钮 + 大圆角药丸
- emoji 当图标用
- 假数据 / 假 logo / 假"X 万用户"
- 整章 N 步用同一种入场动画（全场 fade / 全场 blur）
- 每步都挂 ken burns / 光晕呼吸 / 持续闪烁
- 每屏右下角都挂 mono 角标 / 序号

缺的东西**承认缺** —— 用 placeholder 占位卡（一张写着"image · 16:9
描述"的卡片，按真实比例留位）。**不要**用 emoji 凑、不要找无关图凑、
不要编数字。**没有就承认没有**，比 fake 强一百倍。

## 框架已经搭好的部分（理解就好，不需重写）

- **16:9 固定舞台**：内容设计在 1920×1080 上，外层 transform scale
  缩到任何视口，外围 letterbox 留黑 —— **没有响应式断点**
- **舞台居中 + 大留白**：上下左右四边都让出至少 80px 的安全区
- **隐形进度条**：屏幕底部默认完全透明，鼠标悬到底部边缘才出现，
  支持点击跳转章节（放映 / 投屏时画面上看不到任何 chrome 控件）
- **全局 step 驱动**：点击舞台空白处 / 键盘 ←/→ 推进；章节是 `step`
  的纯函数，没有定时器、没有命令式状态

## 代码层最小约束

不能踩的红线，其它怎么写都行：

### 必须用 token（换主题不破的底线）

- **颜色**：`--shell` / `--surface` / `--surface-2` / `--surface-3` /
  `--text` / `--text-2` / `--text-mute` / `--text-faint` / `--rule` /
  `--accent` / `--accent-soft` / `--accent-glow` ——
  **禁硬编码 hex / rgb / 颜色名**
- **字体家族**：`--font-display-cn` / `--font-display-en` / `--font-body`
  / `--font-mono` —— **禁硬编码字体名**
- **主题性格签名**通过 primitive class 自动接入，**不要在章节 CSS 里
  重定义它们**：
  - `.hero-num`（hero 数字风格 —— 主题决定衬线 / 等宽 / 粗黑）
  - `.rule`（分割线 —— 主题决定 1px 实线 / 4px 实线 / 2px 虚线）
  - `.card`（卡片 —— 主题决定圆角 + 阴影性格）
  - `.stage-frame`（舞台底色 / 圆角 / 阴影 / 装饰图案 / vignette
    全自动，章节什么都不用做）

### 可硬编码 / 可 token，按内容自由（解锁章节自由设计）

- **字号**：想要 80px 就写 80px，想用 `var(--t-h1)` 也行
- **间距 / padding / margin**：按画面节奏写具体值
- **动画时长 / 缓动 / keyframe**：按动画意图写具体值
  （**节奏气质**参考 `theme.json` 的 `mood` —— 慢主题别写 200ms 的快动画）
- **边框宽度 / 非性格圆角 / 字距**：随手写
- **gap / grid 布局尺寸**：按画面构图写

### 其它工程红线

- 不用 `setTimeout` / `setInterval` 驱动动画 —— 用 CSS keyframes
- 章节内的可交互元素（按钮 / 自定义控件）加 `data-no-advance`，
  否则点了会被舞台误推进 step
- 章节代码物理隔离：每章独立文件夹、独立 CSS 类前缀，不跨章 import
- **每章必须有 `notes.ts`**（与 `<Chapter>.tsx` 同目录）：
  - `export const notes: StepNote[]`（`StepNote = string`）
  - 数组长度 **=** 章节代码里 `if (step === N)` 出现的最大 N + 1
  - 每个元素 = 一个 string，该 step 的**演讲备注 / 讲解提词**（写给
    讲者自己看的），放映时按 `N` 键开 / 关浮层显示，**绝不渲染到 slide
    上**，也**不参与任何音频 / 视频流程**（本 Skill 没有这些）
  - 还没想好提词的 step 先用空串 `""` 占位（数组长度仍要对得上 step 数）
  - 这是**章节 step 数的唯一真相源**——`notes.length` 与代码里 step 分支
    一旦对不上，放映时翻页计数就会错位
- **每步动画的视觉节奏由 step 本身决定**，不绑定任何音频长度（本 Skill
  没有音频）。动画"出现得干净、停下不抢戏"即可——详见下方
  「时长参考」与 [`THEMES.md`](THEMES.md) 的 `mood` 气质。

## 时长参考（视觉节奏，**不绑定任何音频**）

放映是**手动推进**的：讲者讲完一个 step 才点下一步。所以这里给的不是
"必须卡到几秒"的硬时长，而是**一个 step 的入场动画在屏幕上"感觉应该
多长"**的视觉节奏参考——目标是「**讲者讲到这一步时，动画刚好已经落定、
不抢话也不让人空等**」。

- **核心原则 = 一步一个想法**：step 的信息密度决定它在屏幕上停留多久，
  而不是反过来。一个 step 挂了 1~3 个东西就讲一阵、好几个并列项就该
  **拆成多个 step**（见上方「逻辑节拍 = step」）。
- **入场动画"瞬间落定"的体感参考**（仅气质，不是验收红线）：

  | 节奏气质（看 `theme.json` 的 `mood`） | 单个主导动作落定的体感 |
  |---|---|
  | 利落 / punchy（terminal / bauhaus / swiss） | ~0.2~0.5s 干脆出现 |
  | 中速 / springy（keynote / pastel / zine） | ~0.5~0.9s 带一点 overshoot |
  | 电影感慢镜（midnight-press / dune / 文献感主题） | ~1.2~1.8s 慢锐化 / 慢推 |

- **动画落定后要"稳得住"**：讲者还在讲这一步时，画面不能一直在动到
  抢戏。持续微动（呼吸 / 漂移）**按需挂、能省则省**（见「避免 AI 味」
  关于"每步都挂 ken burns / 持续闪烁"的反模式）。
- **不写死毫秒进 outline**：`outline.md` 只规划节拍与信息密度，具体
  ms / 缓动 / keyframe 由章节实现时按 `mood` 拍板（见
  [`OUTLINE-FORMAT.md`](OUTLINE-FORMAT.md)）。

## 完工自检（写完每章**强制**执行，不可跳过）

> ⚠️ **硬性流程**：章节实现完成后**必须**走完下面的自检 → 修复 → 汇报
> 三步。**禁止**"实现完成 → 直接汇报给用户"。
>
> **执行方式**（按能力降级）：
>
> 1. **优先 Agent Teams**：开一个独立的 reviewer agent，传入本章代码路径
>    + 本文件 Part「完工自检」清单，让它**逐项核查 + 出结论**（哪几条
>    pass / 哪几条 fail + 证据）。
> 2. **其次 subAgent**：当前 agent 没有 Teams 能力但能开 subagent，用 subagent
>    走同样的流程。
> 3. **都没有**：当前 agent 自己**严格逐项**核查，不允许目测一遍就放行。
>
> **拿到自检结论后**：先按 fail 项**改完代码**，然后再向用户汇报"做完
> 了 + 自检结论 + 改了什么"。**直接拿原始结论汇报但不修复 = 违规**。

写完一章 + 在浏览器点完一遍后逐项过：

- [ ] **每章至少 1~2 处 CSS / SVG / Canvas / JS 视觉演示** —— 没有 = 回去补
- [ ] **不同 step 的主导动作不一样** —— 全章一种动画 = 回去重做
- [ ] 字号大、留白舒服、配色舒服（🎤 deck 式可收紧字号 / 留白，但配色 / 层级仍要舒服、字号远观可读）
- [ ] 清单 / 列表逐步揭示（🎬 视频式 **1 项 = 1 step**；🎤 deck 式 = 同一页逐步 build）
- [ ] 画面信息比 outline 字面多（回了原始素材抽细节挂上来）
- [ ] 没有紫粉渐变 / 圆角彩色边框 / emoji / 假数据 / 假 logo
- [ ] 缺的素材用 placeholder，不是 fake
- [ ] **颜色和字体家族全部走 token**（无硬编码 hex / 字体名）；hero 数字
      / 卡片 / 分割线 / 舞台用 primitive class 接入主题性格 —— 这两条不
      达标 = 换主题就破
- [ ] 章节交付时**主动告诉用户**："本章还缺这些素材"
- [ ] 禁止字小到看不清、禁止纯文字墙（🎤 deck 式信息可多，但仍须有视觉演示、字号远观可读）
- [ ] 禁止出现任何形式的页眉页脚，仅展示关键内容（出现后必须回去改）
- [ ] **`npx tsc --noEmit` 通过** —— 不通过禁止汇报"做完了"
- [ ] 章节代码物理隔离：独立 CSS 类前缀（`.cd-` / `.mg-` / ...），
      未跨章 import，未修改 `chapters.ts` 之外的共享文件
- [ ] **`notes.ts` 存在**且 `export const notes: StepNote[]` 的
      `notes.length` === 章节代码里 `if (step === N)` 用到的最大 N + 1
      （不一致 = 放映翻页计数错位；`notes.length` 是 step 数唯一真相源）
- [ ] **每条 note 是给讲者的提词**（讲解要点 / 提醒），按 `N` 键看得到，
      **绝不出现在 slide 上**，也不参与音频 / 视频（本 Skill 没有）
- [ ] **每步动画"出现即落定、停下不抢戏"**，视觉节奏与 step 信息密度
      匹配（参「时长参考」）—— 没有任何"必须卡音频长度"的约束

任一未过 → 回去改。**不要**"先放着以后修"。
